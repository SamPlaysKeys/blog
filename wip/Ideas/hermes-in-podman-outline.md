# Blog Outline: Running Hermes AI Assistant in a Podman Container

*A step-by-step guide to containerizing a Discord-integrated AI assistant with Podman's daemonless, rootless architecture.*

---

## Outline

### 1. Title & Hook
- Working title: *"Hermes in a Pod: Running an AI Assistant with Podman"*
- Hook: Your AI assistant shouldn't need a massive daemon just to run in a container. Podman's fork/exec model is a natural fit for lightweight, persistent agent services.

### 2. What is Hermes?
- A persistent, Discord-integrated AI assistant for brainstorming, research, documentation
- Dual-agent architecture: Hermes handles ideation (OpenCode handles coding)
- Uses cloud APIs (OpenAI, Anthropic, Together AI) for inference
- Optional: local models via Ollama if hardware supports it
- Features: web search, RAG via vector DB, markdown generation, conversation memory

### 3. Why Podman Over Docker for Hermes?

| Concern | Docker | Podman |
|---|---|---|
| Daemon footprint | `dockerd` always running | No daemon — systemd socket activation |
| Rootless | Opt-in, less mature | Default — Hermes never has root |
| Systemd integration | Workaround needed | `podman generate systemd` — native |
| Pod semantics | Extra tooling | Native `pod play kube` |
| Restart policy | Docker restart policies | systemd-managed — survivable |

- For a service expected to run 24/7 (Hermes), Podman's systemd-native restarts and rootless default align better than Docker's daemon-centric model.

### 4. Prerequisites
- Podman 4.5+ (or Podman Desktop on macOS)
- Discord bot token and application setup
- API keys (OpenAI, Anthropic, etc.)
- Optional: Ollama for local inference
- Optional: Vector DB (ChromaDB, Qdrant) for RAG memory

### 5. Container Setup

#### 5a. Building the Hermes Image
- Dockerfile structure (multi-stage: python base + dependencies)
- `Containerfile` vs `Dockerfile` — Podman handles both identically
- Build with `podman build -t hermes:latest .`

#### 5b. Podman-Specific Configuration
- Rootless considerations: port mapping (>1024), volume ownership
- User namespace remapping for volume permissions
- SELinux/AppArmor (Linux) — Podman works with system LSM by default
- seccomp profiles for API call restrictions

#### 5c. Running the Container
```bash
podman run -d \
  --name hermes \
  --replace \
  --restart on-failure \
  -v ./data:/app/data:Z \
  -e DISCORD_TOKEN=... \
  -e OPENAI_API_KEY=... \
  hermes:latest
```
- Compare with equivalent `docker run` — nearly identical syntax
- Note the `:Z` label for SELinux contexts (Linux only)

### 6. Systemd Integration (The Killer Feature)
- Generate systemd unit: `podman generate systemd --name hermes --files`
- Install to systemd: copy to `~/.config/systemd/user/`
- Enable and start: `systemctl --user enable --now container-hermes`
- Survives reboots, user logouts, and crashes
- Logs via `journalctl --user -u container-hermes`
- This is where Podman fundamentally outshines Docker for long-running services

### 7. Podman Quadlets (Alternative to Systemd Generation)
- More declarative approach: define containers via `.container` files in `~/.config/containers/systemd/`
- Systemd units auto-generated by systemd-sysext
- Hermes Quadlet example:
```yaml
[Container]
Image=hermes:latest
Volume=./data:/app/data
Environment=...
```
- Arguably cleaner than systemd unit files for container-only workloads

### 8. Podman Machine on macOS
- `podman machine init` creates a Fedora CoreOS VM
- `podman machine start` — Hermes runs in the VM, transparent to host
- Volume mounts: `~` or `/Users` paths are automatically shared
- Port forwarding: `--userns=keep-id` for file permissions
- Systemd integration in the VM — Hermes persists across machine restarts

### 9. RAG and Memory with Podman Pods
- Define a pod containing Hermes + ChromaDB + Redis
  ```bash
  podman pod create --name hermes-stack -p 8000:8000
  podman run --pod hermes-stack -d --name chroma chromadb/chroma
  podman run --pod hermes-stack -d --name hermes hermes:latest
  ```
- Pods share network namespace — Hermes talks to ChromaDB at `localhost`
- Alternative: `pod play kube` from a Kubernetes YAML for portability

### 10. Production Considerations
- Resource limits: `--memory`, `--cpus`, `--pids-limit`
- Health checks: `--health-cmd`, `--health-interval` — Podman supports OCI health checks natively
- Secrets management: `podman secret create` or environment files with proper permissions
- Logging: `journald` driver for systemd integration, or `k8s-file` for structured logs

### 11. Comparison: Same Setup with Docker Compose
- Hermes as a `docker-compose.yml` service
- Pros: simpler mental model, better tooling for multi-container stacks
- Cons: no systemd integration, daemon-dependent restart, rootful by default
- When Compose still makes sense (teams already invested, CI parity)

### 12. The Verdict
- Podman is the better runtime for persistent AI agent services — systemd-native restarts, rootless, no daemon tax
- The learning curve is shallow: `alias docker=podman` works for 95% of Hermes' operations
- For a 24/7 assistant like Hermes, the systemd integration alone justifies the switch
- macOS users pay a small VM tax, but get the same reliability benefits on the Linux side
